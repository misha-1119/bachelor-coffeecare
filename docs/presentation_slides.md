## Slide 5 — Архітектура

**Тип:** клієнт-серверна, моноліт на Python.

**Потік користувача:**

```
[Лендинг CoffeeCare (Next.js, Vercel)]
              │
              ▼ (CTA → t.me/CaffeBot)
[Telegram-клієнт користувача]
              │
              ▼ (long-polling)
[Backend на Python: python-telegram-bot v21]
   ├── Qdrant (локально, file-backed) — векторний пошук
   ├── Convex (cloud) — KB та профілі користувачів
   └── Ollama HTTP — локальний LLM (Lapa, liberta)
```

- **Лендинг** = маркетинговий вхід, статика на Vercel. **Не звертається до бекенду напряму** — переводить користувача в Telegram.
- **Telegram-бот** = єдиний фронт для діалогу. Уся бізнес-логіка та AI на бекенді.
- **AI + Backend** = retrieval (Qdrant) + Convex + LLM (Ollama) — дають відповідь.

**Чому не мікросервіси:** QPS низький, найдорожчий компонент (LLM) не паралелиться розщепленням; додаткова мережева латентність зайва. Лендинг винесено окремо лише через інший SLA (статичний edge-кеш).

## Slide 6 — AI-пайплайн (RAG)

1. **Triage** — швидкі правила (привітання, прощання, «детальніше», safety-keywords). Закриває ~20% запитів без виклику моделей.
2. **Кодування запиту** — український BERT `liberta-large`, 1024-вимірний вектор.
3. **Qdrant retrieval** — HNSW, top-50 кандидатів.
4. **Re-rank** — cosine + keyword boost.
5. **Поріг впевненості:**
   - **≥ 0.55** → генератор `Lapa` (Ollama) переписує відповідь українською у розмовний стиль (2–4 речення).
   - **< 0.55** → каскадний fallback по PDF-чанках: модель → бренд → універсал.
6. **Regex-override для кодів помилок** — спрацьовує до векторного пошуку, +12% точності на запитах з друкарськими помилками.

## Slide 7 — Технології (обґрунтування вибору)

| Шар | Технологія | Чому саме вона |
|---|---|---|
| Embedder | **liberta-large** | Domain-українська лексика; multilingual-MiniLM програє на термінах. |
| Vector DB | **Qdrant** (file-backed) | Payload-фільтри для каскаду по моделі/бренду; FAISS це не вміє. |
| Генератор | **Lapa** (Ollama) | Український токенізатор; нульова варіативна вартість; локальність даних. |
| LLM runtime | **Ollama** | vLLM на Apple Metal нестабільний; Ollama дає той самий HTTP-інтерфейс. |
| Cloud DB | **Convex** | Live-оновлення KB без редеплою бота. |
| Bot framework | **python-telegram-bot v21** | Async-нативний, long-polling без webhook-інфраструктури. |
| Frontend | **Next.js 15 + React 19** на Vercel | Ізольований від ML-стеку, окремий deploy-цикл. |

> Жодного вибору на основі «найпопулярніше» — кожен компонент має конкретний технічний компроміс.

**Дві моделі монетизації:**

### B2B — продаж ліцензії бізнесу

- **Хто:** ресторани, кав'ярні, готелі, дистриб'ютори кавомашин, мережі сервіс-центрів.
- **Що отримує:** доступ до бота 24/7 для самостійного усунення несправностей власних кавомашин без виклику майстра.
- **Формат:** платіж за ліцензію (помісячна підписка або річний контракт), без обмеження кількості запитів.
- **Метрика цінності:** % звернень, закритих без виклику майстра; економія 20–40% на сервісних викликах.

### B2C — пряма монетизація запитів

- **Хто:** кінцеві користувачі-власники кавомашин вдома.
- **Формат:** **pay-as-you-go** — оплата за кожен діалог (або пакет з N запитів).
- **Чому не freemium:** немає сенсу віддавати запити безкоштовно — продукт вирішує конкретну болючу проблему, користувач готовий платити за разовий результат замість виклику майстра за 800–1500 грн.

## Slide 12 — Оцінка економічної ефективності

> Розрахунки за методикою ІТ СТЕП (Пшик Б.І., 2024). Норма дисконту r = 15%, горизонт 4 роки.

**Заплановані показники до підрахунку:**

1. **NPV** — чиста теперішня вартість грошових потоків за 4 роки.
2. **PI** — індекс дохідності (∑дисконтованих CF / інвестиції).
3. **B/C** — коефіцієнт «вигоди / витрати».
4. **DPP** — дисконтований період окупності.
5. **Точка беззбитковості** — Q\* = FC / (P − VC).

**Структура витрат для точки беззбитковості:**

- **Постійні (FC):** Convex, Vercel, домен, амортизація обладнання, маркетинг (паушально).
- **Змінні (VC):** обчислення на запит — електроенергія LLM, токени, трафік. Орієнтовно ≈ 0.005 у.о. / діалог.
- **Ціна (P):** B2B — фіксована ліцензія; B2C — фіксована ставка за запит.

> Конкретні значення NPV, PI, B/C, DPP, Q\* будуть розраховані у фінальній економічній частині після фіналізації прогнозу клієнтської бази (B2B-контракти + B2C-обсяг запитів).

## Slide 13 — Точка беззбитковості (план розрахунку)

**Що буде розраховано:**

1. **Постійні витрати (FC)** на рік — Convex+Vercel+домен+амортизація+маркетинг.
2. **Змінні витрати на одиницю (v)** — собівартість одного діалогу (інференс LLM + retrieval).
3. **Ціна одиниці (P)** — B2B-ліцензія / місяць та B2C — ціна за запит.
4. **Q\* (точка беззбитковості)** за формулою:

   ```
   Q* = FC / (P − v)
   ```

5. **Графік беззбитковості** — перетин ліній TR(Q) та TC(Q); візуалізація зони прибутку / збитку.

**Очікуваний результат:** перевірити, чи перші B2B-контракти на пілотному етапі (1–2 клієнти) вже виводять проєкт у зону прибутку, та оцінити мінімальний обсяг B2C-запитів за відсутності B2B-канала.

## Slide 14 — Діагностика рентабельності (план)

**Заплановані показники рентабельності (за методикою ІТ СТЕП, табл. 15):**

1. **ROS — Рентабельність продажу** = Валовий прибуток / Виручка
   - Порогові: <6% дуже низька · 6–21% середня · 21–31% висока · >31% дуже висока.
   - Очікувана позиція CaffeBot: **середній / високий рівень** (низькі змінні витрати на запит).
2. **ROI — Рентабельність інвестицій** = Чистий прибуток / Початкові інвестиції
   - Тенденція: зростання, порівняння з альтернативними проєктами.
3. **Рентабельність витрат** = Прибуток до оподаткування / Сукупні витрати
   - Цільове: ≥ 20%.

> На попередньому захисті — методика і порогові значення. Фактичні числа — у фінальній економічній частині після уточнення прогнозу продажів.
